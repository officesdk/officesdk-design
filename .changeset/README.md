# Changesets

Hello and welcome! This folder has been automatically generated by `@changesets/cli`, a build tool that works with multi-package repos, or single-package repos to help you version and publish your code. You can find the full documentation for it [in our repository](https://github.com/changesets/changesets)

We have a quick list of common questions to get you started engaging with this project in [our documentation](https://github.com/changesets/changesets/blob/main/docs/common-questions.md)

## How to Use Changesets

### Adding a Changeset

When you make changes that affect users, run:

```bash
yarn changeset
```

Follow the prompts:

1. Select `@officesdk/design` (the only package we publish)
2. Choose version bump type:
   - **patch** (0.1.0 to 0.1.1) - Bug fixes
   - **minor** (0.1.0 to 0.2.0) - New features
   - **major** (0.1.0 to 1.0.0) - Breaking changes
3. Write a clear summary

This creates a markdown file in `.changeset/` describing your change.

### Example Changeset

```markdown
---
'@officesdk/design': minor
---

Add Dropdown component with the following features:

- Single and multiple selection support
- Keyboard navigation
- Search functionality
- Custom trigger element
```

### When to Add a Changeset

**DO add a changeset for**:

- [YES] New components
- [YES] New features
- [YES] Bug fixes
- [YES] API changes
- [YES] Performance improvements (user-visible)

**DON'T add a changeset for**:

- [NO] Documentation updates
- [NO] Test updates
- [NO] Internal refactoring (no API change)
- [NO] CI/CD configuration
- [NO] Development tooling

### Release Process

1. **Developer**: Adds changeset with PR
2. **CI**: Validates PR (lint, test, build)
3. **Merge**: PR merged to main
4. **Changesets Bot**: Creates "Version Packages" PR automatically
5. **Maintainer**: Reviews and merges version PR
6. **GitHub Actions**: Automatically publishes to NPM

### Version Bump Guidelines

#### Patch (0.1.0 to 0.1.1)

- Bug fixes
- Documentation updates
- Minor improvements
- No new features
- No breaking changes

#### Minor (0.1.0 to 0.2.0)

- New features
- New components
- New props (backward compatible)
- Deprecations (with warnings)

#### Major (0.1.0 to 1.0.0)

- Breaking API changes
- Removed features
- Renamed props
- Changed behavior
- Removed components

### Tips

1. **Be descriptive**: Write clear changeset summaries
2. **One changeset per PR**: Usually one changeset is enough
3. **Multiple changes**: If your PR has multiple unrelated changes, consider splitting it
4. **Breaking changes**: Always document migration path

### Checking Changesets

```bash
# List all changesets
ls .changeset/*.md

# View a specific changeset
cat .changeset/your-changeset-name.md

# Remove a changeset (if needed)
rm .changeset/your-changeset-name.md
```
